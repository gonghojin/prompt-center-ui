# 프롬프트 수정(소프트 업데이트) 기능 작업 계획

## 1. 요구사항 및 정책 요약

- 프롬프트(PromptTemplate)는 수정 시 PromptVersion에 EDIT 이력이 남아야 함
- 작성자/권한자만 수정 가능, 수정 시 새 버전 생성
- 상태/가시성/태그/변수 등 주요 필드 수정 가능
- API: PUT /api/v1/prompts/{uuid} (예상)
- 예외: 권한 없음(403), 미존재(404), 유효성 오류(400)

## 2. 화면/UX 설계

- `/prompts/[id]/edit` 경로에 프롬프트 수정 페이지 구현
- 주요 입력 필드: 제목, 설명, 카테고리(드롭다운), 태그(동적 추가/제거, 중복 방지), 변수, 상태, 가시성 등
- 초기값: GET /api/v1/prompts/{uuid}로 상세 조회 후 폼에 바인딩
- 태그: Enter/버튼으로 추가, X로 제거, 중복 방지
- 상태/가시성: 드롭다운/토글로 선택, 변경 시 UI에 반영
- 임시저장/저장하고 게시 버튼 제공
- 저장/게시 성공 시 상세/목록 갱신, 실패 시 에러 안내
- 권한/예외: 작성자/권한자만 수정 가능, 403/404/400 등 안내

## 3. API 연동 및 데이터 흐름

- GET /api/v1/prompts/{uuid} → 프롬프트 상세 조회(초기값)
- PUT /api/v1/prompts/{uuid}
    - payload: { title, description, categoryId, tags, inputVariables, visibility, status }
    - 임시저장: status=DRAFT, actionType=EDIT
    - 저장하고 게시: status=PUBLISHED, actionType=PUBLISH
    - 성공: 수정된 프롬프트 정보(새 버전)
    - 실패: 400/403/404 등

## 4. 예외/권한 처리

- 403: "수정 권한이 없습니다" 안내, 수정 불가
- 404: "존재하지 않는 프롬프트" 안내
- 400: 입력값 오류 안내
- 로딩/에러/성공 상태 UI에 명확히 표시

## 5. 컴포넌트/로직 구조

- `/pages/prompts/[id]/edit.tsx`
    - useEffect: 프롬프트 상세 조회(초기값)
    - `<PromptEditForm />`: 수정 폼 컴포넌트
        - 제목, 설명, 카테고리, 태그, 변수, 상태, 가시성 등 입력
        - 태그 추가/제거 로직(중복 방지)
        - 임시저장/저장하고 게시 버튼
        - onSubmit: PUT /api/v1/prompts/{uuid} 호출
        - 성공/실패/로딩/권한 UI 처리

## 6. UI/UX 흐름

1. PromptCard의 "편집" 클릭 → `/prompts/[id]/edit` 이동
2. 수정 폼: 기존 값 표시, 입력/수정
3. 태그: 동적 추가/제거, 중복 방지
4. 상태/가시성: 드롭다운/토글로 변경
5. 임시저장/저장하고 게시 버튼
6. API 호출: PUT, 성공 시 상세/목록 갱신, 실패 시 에러 안내
7. 권한/예외: 안내 메시지

## 7. 추가 고려사항

- 버전 이력: 수정 시 새 버전(EDIT) 생성(백엔드 처리)
- 상태/가시성/태그 등 모든 주요 필드 수정 가능
- 권한/유효성/예외: 명확한 안내
- 코드 품질: Tailwind, 타입, 접근성, 가독성

3. **프롬프트 상세 조회 API 연동**
    - GET /api/v1/prompts/{uuid}로 초기 데이터 패칭 및 폼 바인딩

6. **수정 API 연동 및 예외 처리**
    - PUT /api/v1/prompts/{uuid} 호출, 성공/실패/권한/로딩 UI 처리

7. **상세/목록/버전 이력 UI 갱신**
